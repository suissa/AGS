# 05 — Project Sync Rules Engine

Implemente o Project Sync Rules Engine.

## Objetivo

Transformar eventos em transições de estado nos GitHub Projects.

```txt
EVENT -> RULE -> ACTION
```

## Eventos mínimos

```txt
task.created
branch.created
commit.pushed
pr.opened
pr.merged
ci.started
ci.success
ci.failed
llm.used
task.converged
dependabot.alert
security.alert
```

## Ações mínimas

```txt
add_to_project
move_project_item
set_field
create_issue_comment
create_bug_issue
create_security_issue
attach_ctid
write_trace
```

## Arquivos esperados

```txt
src/sync/project-sync-rules-engine.ts
src/sync/rule-types.ts
src/sync/default-rules.simple.ts
src/sync/default-rules.allascode.ts
src/sync/rule-executor.ts
tests/project-sync-rules-engine.test.ts
```

## Regras simple

- `task.created` -> add to Execution
- `pr.opened` -> status In Review
- `ci.failed` -> status Blocked
- `pr.merged` -> status Done

## Regras allascode

- `task.created` -> Backlog
- `branch.created` -> Execution / In Progress
- `pr.opened` -> Execution / In Review
- `ci.failed` -> Bug Tracker + Execution / Blocked
- `llm.used` -> Cognitive
- `dependabot.alert` -> Security
- `pr.merged` -> Execution / Done
- `task.converged` -> Cognitive + Wiki snapshot

## Critérios de aceite

- Engine executa regras por evento.
- Regras são configuráveis.
- Engine testável sem GitHub real.
- Logs para TUI.


## Regras gerais para a IA implementadora

- Implemente apenas o escopo deste arquivo.
- Crie ou atualize testes automatizados.
- Atualize documentação quando necessário.
- Não persista chain-of-thought bruto; use semantic traces estruturados.
- Não avance para a próxima etapa sem validação local.
- Use TypeScript/Node.js salvo se o projeto já tiver outra stack explicitamente definida.
- Nunca hardcode tokens, API keys ou secrets.
- Todo módulo deve funcionar em dry-run quando depender de GitHub real.
